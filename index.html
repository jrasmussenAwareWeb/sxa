Helix-Habitat Architecture


The most difficult concept for me to grasp while getting to know SxA is that the actual design happens outside of Sitecore via the Creative Exchange.
Although I understand the ultimate benefit, wrapping my brain around what that means to a development workflow process has been ... an exercise, to say the least.
To understand why this is important, let's first get a better understanding of the underlying Habitat modular architecture framework built on Helix principles.


What is Habitat?

Sitecore Habitat is a real Sitecore project implemented on the Sitecore Experience Platform using Helix and is an example that allows developers to see how Helix
is applied and lets developers experience a project based on these principles. In addition, it saves the developer from having to create these modules from scratch
and also provides an excellent basis on which to develop additional modules and extend existing ones.


What is Helix?

Helix is a set of overall design principles and conventions for Sitecore development often referred to as Component-based Architecture or Modular Architecture.
In short, Helix and Habitat are built upon a set of principles and conventions focused on simplicity, flexibility, and extensibility.


What is Modular Architecture?

Modular Architecture is an architectural approach that was formalized for Sitecore solutions at Pentia where it has been in use since around 2009 and is focused on simplifying
the development of the solution by breaking the solution into smaller independent and manageable parts. The Architecture in Habitat is primarily derived from the
Package Principles defined by Robert C. Martin, which describes how to avoid dependencies between unrelated parts by applying simple patterns.
 

Layered Architecture

Note that the layers in Modular Architecture are not equivalent to the layers seen in 3/n-tiers architecture even though they bear resemblance in terms of dependency direction.

 

Project (or Context) layer

This layer provide the context of the solution. This means actual pagetypes, layout and design. It is on this layer where all the features of the solution is tailored together.
Each time there is a change in a feature, a new one is added or one is removed, then this layer will change. The layer also contains the concrete design of the solution.
This means that in terms of the Stable Dependencies Principle then the modules on this layer are unstable - the most unstable modules in your solution.

 

Domain (or Features) layer

The Feature layer is the functionalities and the features of the website such as navigation, news, page contents, etc. all the presentation components and business logic
should be here in this layer. The domain layer contain concrete features of the solution as understood by the editors and owners of the solution.
The features are expressed as seen in the domain of the solution and not by technology.

Each Domain layer module has to strictly conform with the Common Closure principle. What changes together should live together alone.
This ensures that changes in one feature does not cause changes anywhere else. Features can be added modified and removed without impacting other features.
Strict layering within the Domain layer is very important. One Domain module must never depend on another Domain module.
Breaking this will make you lose many of the benefits that Modular Architecture provides.

 

Framework (or Foundation) layer

The foundation layer contains the services and the back end of the project that are used all over the other layers.
This layer contain modules that is the foundation of your solution. When a change occur in one of these modules it will impact other modules in the solution.
The modules are conceptually abstract and does not contain presentation in the form of renderings since these are to be considered concrete.
There is not Strict Layering on the framework layer. This means that one framework module can depend on another framework module in the solution.

 

Technology Layer

Finally, the above layers sit on top of the technology stack.



 

What does this mean to us? 

  1. Keep HTML semantic
  2. Avoid loose coupling as much as possible
  3. Keep track of dependencies and keep references internal
  4. Create common functionality framework modules
